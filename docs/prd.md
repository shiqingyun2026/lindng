## 1. 产品概述



### 1.1 项目背景

随着家长对儿童体能发展的重视，少儿体适能培训需求旺盛，但现有机构多需固定场地接送，耗时且成本高。深圳市场调研显示，家长对“家门口”的便利课程及拼团降价模式有强烈需求。本产品旨在连接社区家长与体适能教练，通过小区拼团、上门授课的方式，降低课程价格，解决接送难题。



### 1.2 产品定位

一款面向深圳社区家长的微信小程序，让家长在小区内发起或参与儿童体适能课程拼团，平台负责课程配置与教练协调，实现“满员即开、未成团退款”的轻量化服务。



### 1.3 核心目标

* MVP验证家长对社区拼团模式的接受度与复购率

* 跑通“运营配置课程→家长开团/参团→自动成团→线下授课”的业务闭环

* 积累种子用户，为后续迭代提供依据



### 1.4 核心规则说明

* **一个课程同一时刻仅允许一个进行中的拼团**：课程支持配置最大成团次数（`max_groups`），但任一时刻只允许存在一个进行中的拼团。当前进行中的拼团结束后，若未达到课程开团上限，后续用户仍可继续为该课程发起新团。

* **无评价功能**：MVP版本不包含课后评价，相关页面已移除。

* **拼团规则透明**：在课程详情页和支付页均展示拼团规则，明确用户权益。

* **首次启动先同意协议**：用户首次进入小程序时，需先阅读并同意《用户协议》《隐私政策》后方可进入首页，拒绝则无法继续使用。

* **同一用户同一课程仅可成功参团一次**：用户一旦成功参加过某课程的任一拼团，不可重复参加该课程后续拼团，也不可再次为该课程开团。

* **未支付不算参团成功**：创建订单但未支付时，不增加拼团人数，不进入“我的拼团”，仅在支付成功后写入正式参团关系。

* **课程人数展示按页面场景区分**：首页课程卡片展示课程历史累计成功拼团人数，并在存在进行中的拼团时额外展示该团倒计时；课程详情页若课程只允许开1个团，可展示累计成功人数，若允许多团则不在顶部展示“已拼X人”，改在拼团模块中展示所有团状态；拼团详情页保持单团详情定位。



## 2. 目标用户与用户画像



### 2.1 目标用户群

* **职场妈妈**（30-40岁，双职工家庭，孩子3-10岁）：追求便利，节省接送时间，信任邻居推荐

* **全职妈妈**（28-38岁，注重孩子全面发展）：有社交需求，乐于组织社区活动，价格敏感但愿意为质量付费

* **带娃长辈**（60-70岁，子女双职工）：操作需简便，极度关注安全，依赖子女协助报名



### 2.2 典型用户画像



**画像1：职场妈妈——李女士（34岁，南山科技园产品经理）**

* 家庭情况：一家三口，儿子5岁，双职工，住南山区某大型小区

* 育儿痛点：工作繁忙，周末想陪孩子但又累；报过商场里的体能课，但接送来回1小时，停车难；孩子体能需加强，但兴趣班太多

* 价格敏感度：中等偏高，愿意为便利和质量付费，希望性价比高；能接受单次80-120元，包月300-500元

* 决策关键：便利性 > 价格 > 教练名气；希望就在小区内，不用开车；对教练资质有一定要求，但更看重孩子是否喜欢



**画像2：全职妈妈——王姐（38岁，福田某高端小区，二孩妈妈）**

* 家庭情况：两个女儿（8岁和4岁），丈夫经常出差，住福田中心区大型社区

* 育儿理念：注重全面发展，女儿们报了钢琴、舞蹈、英语，但户外活动少，想增加体能锻炼

* 价格敏感度：较高收入，但对价格仍理性，希望“物有所值”；愿意为优质课程支付溢价

* 决策关键：课程专业性 + 社交机会；希望课程有体系，教练懂儿童心理；同时希望孩子能跟小区玩伴一起上课，增进友谊



**画像3：年轻全职妈妈——小雅（28岁，龙岗区，宝宝2岁）**

* 家庭情况：一家三口，儿子2岁，丈夫在互联网公司工作，小雅全职带娃，住龙岗大型社区

* 育儿痛点：孩子精力旺盛，每天需要大量户外活动，但自己缺乏专业引导；想报早教班但价格高（200+元/节），且需固定时间地点；希望孩子多和同龄人互动

* 价格敏感度：对价格敏感，期望单次50-80元，包月200-300元；愿意拼团降低费用

* 决策关键：课程适龄性 + 教练亲和力；2岁宝宝需要温柔耐心的教练，课程以游戏为主；场地安全、干净



**画像4：带娃长辈——张爷爷（65岁，罗湖区，孙子6岁）**

* 家庭情况：儿子儿媳双职工，工作日主要由张爷爷接送孙子上学、带孙子户外活动，住罗湖老小区

* 育儿特点：疼爱孙子，但担心自己不懂教育，希望孙子多运动少看电视；平时带孙子在小区广场玩滑板车、追逐

* 数字能力：会用微信发消息、看视频，但不熟悉小程序操作，复杂的报名流程需要子女帮忙

* 决策关键：便利性 + 安全性：就在小区内，不用远走；有保险保障；课程时间在白天（他带孩子时段）；教练可靠

* 特殊需求：希望报名方式简单（如子女代报，或小程序有大字版/语音辅助）；课程提醒及时

## 4. 详细功能需求



### 4.1 家长端小程序



#### 4.1.1 启动协议、登录与授权



**触发场景**：

* 用户首次进入小程序时，需要先完成协议同意。

* 用户主动点击“我的”页面中的登录按钮。



**交互流程**：

1. **首次进入小程序**：

   * 小程序启动后，优先检测用户是否已同意协议。

   * 若未同意，则进入独立协议页；用户必须勾选“我已阅读并同意”后，点击“同意并进入”才能进入首页。

   * 若用户点击“拒绝”，则弹出提示“您需要同意协议才能使用本小程序”；确认后若环境支持则退出小程序，否则停留在协议页。

   * 协议同意状态需本地持久化，已同意用户下次启动不再重复展示。

2. **用户登录**：

   * 用户可在未登录状态浏览首页、课程详情等公开页面。

   * 在“我的”页面点击“点击登录”后，调用 `wx.getUserProfile` 获取用户昵称和头像。

   * 前端再通过 `wx.login` 获取 `code`，请求后端登录接口换取 token，并同步更新全局用户信息与本地缓存。

   * 若用户拒绝授权头像昵称，则停留在当前页并给出友好提示。

3) **登录成功后**：

   * 后端返回token，前端存储，并更新用户状态。

   * 用户头像和昵称在“我的”页面展示。



**数据源**：

* 协议内容：静态富文本，MVP版本直接内置在小程序代码中，后续可由CMS管理。



#### 4.1.2 首页



* **顶部定位栏**：

  * 左侧显示当前定位（如“深圳南山区”），带下拉箭头图标。

  * 点击定位栏触发重新定位，调用微信的 `wx.chooseLocation` 或 `wx.getLocation` 获取新位置，更新列表距离排序。

* **顶部tab**：

  * 全部课程（默认选中）：按距离从近到远显示课程

  * 最近开课：按上课时间从近到远显示课程

* **内容区**：

  * 课程卡片列表，无限滚动加载

* **课程卡片元素及数据来源**：



| 卡片元素           | 数据来源（数据库字段/计算逻辑）                                                     |
| -------------- | -------------------------------------------------------------------- |
| 封面图            | `course.cover`                                                       |
| 课程名称           | `course.name`                                                        |
| 上课时间           | `course.start_time`，格式化为“MM月DD日 周X HH:MM”                            |
| 上课地点           | `course.address`                                                     |
| 拼团价            | `course.group_price`（单位：分，前端转换为元）                                    |
| 原价             | `course.original_price`（划线）                                          |
| 已拼人数           | 显示该课程所有成功拼团的累计成功人数（累计所有 `status='success'` 拼团的支付成功人数） |
| 即将成团标签         | 若该课程存在进行中的拼团且 `current_count >= target_count - 2`，则显示                |
| 剩余时间           | 仅当该课程当前存在进行中的拼团时显示，取该进行中拼团的 `expire_time` 与当前时间的差值，展示为“剩余X天X小时 / 剩余X小时 / 剩余X分钟” |
| 距离（仅“全部课程”tab） | 根据定位或默认坐标计算后展示 `distanceText`                                        |



* **交互**：

  * 点击卡片任意区域进入课程详情页

  * 下拉刷新列表，上拉加载更多

  * 首页课程列表不再默认回退 mock 数据；若后端异常，则直接提示加载失败，便于联调定位



#### 4.1.3 课程详情页



* **页面结构**（从上到下）：

  1. **顶部轮播图**：高度200px，可左右滑动，支持点击放大。数据来源：`course.images`（JSON数组）

  2. **课程信息卡片**（白色背景，圆角，内边距16px）：

     * 课程标题（20px加粗）：`course.name`

     * 价格行：拼团价（24px橙色） 原价（16px灰色划线）：`course.group_price`、`course.original_price`

     * 成团人数：x人成团（图标+文字）：优先取当前进行中的拼团 `activeGroup.targetCount`

     * 已拼人数：若课程仅允许开1个团，可展示该课程历史成功拼团的累计成功人数；若课程允许开多个团，则此处不展示“已拼X人”

     * 上课时间、上课地点、适用年龄（每行图标+文字）：`course.start_time`、`course.end_time`、`course.address`、`course.age_range`

  3) **课程介绍区**：标题“课程介绍”，下方富文本（图片+文字）：`course.description`

  4) **拼团规则说明区**（新增，位于教练信息之前）：

     * 标题：“拼团规则”（加粗，16px）

     * 内容以列表形式展示，每条前带圆点或信息图标，文案如下：

       * 每个课程仅有一个进行中的拼团，先到先得。

       * 达到成团人数自动成团，未达到则自动原路退款。

       * 拼团进行中时可邀请好友加入，人数满额后自动成团。

       * 拼团成功后不可退款，失败自动退款。

       * 如有疑问，请联系客服。

     * 背景色可浅灰，圆角卡片，与教练信息区分隔。

  5. **教练信息区**：

     * 标题“教练信息”

     * 教练姓名（加粗）：`course.coach_name`

     * 教练简介（灰色小字）：`course.coach_intro`

     * 证书图片缩略图（小图，可点击放大）：`course.coach_certificates`（JSON数组）

  6. **保险说明区**：灰色背景区块，文案“本课程赠送运动意外险，由平台统一购买”，带保险图标。固定文案。

  7) **拼团状态区**：

     * 统一展示该课程所有拼团的状态列表，每个团展示：

       * 第几个团

       * 当前状态（进行中/已成团/已结束）

       * 当前人数/成团人数（进度条或数字）

       * 进行中团的剩余时间，已成团或已结束团的状态说明

     * 数据来源：`group`表（`course_id`=当前课程，按最近团优先排序）

  8) **底部固定栏**：

     * 左侧：客服图标（点击弹出客服二维码弹窗）

     * 右侧：唯一主操作按钮，根据课程允许开团数、当前是否存在进行中的团、当前用户是否已参团综合判断



* **特殊状态**：

  * 有进行中的团且用户未参团：按钮显示“去参团”

  * 有进行中的团且用户已参团：按钮显示“已参团，等待成团”，不可点击

  * 无进行中的团且 `completedGroupsCount < maxGroups`：按钮显示“立即开团”

  * 无进行中的团且 `completedGroupsCount >= maxGroups`：按钮显示“已成团”或“课程已满员，暂不可开团”，不可点击



* **交互**：

  * 点击“立即开团”或“去参团” → 进入支付确认页，由后端订单接口根据是否传入 `groupId` 决定开团或参团。

  * 点击客服图标 → 弹出客服二维码弹窗（详见4.1.7）。

  * 点击证书图片 → 放大查看。



#### 4.1.4 开团/参团支付页



* **页面结构**（从上到下）：

  1. **订单信息卡片**：

     * 课程名称：从所选课程传入

     * 上课时间、地点摘要：从所选课程传入

     * 支付金额：¥xx（拼团价，大号字体）：`course.group_price`

  2. **拼团说明卡片**：

     * 成团说明与拼团规则合并为一个统一模块，避免信息重复。

     * 内容包括：

       * 满 \[成团人数] 人成团

       * 拼团截止时间

       * 未成团原路退款规则

       * 简洁版拼团规则说明

  3) **协议勾选区**：

     * 复选框（默认选中）

## 5. 系统规则



### 5.1 拼团规则（核心）

* **一个课程只能有一个进行中的拼团**：每个课程在报名截止时间前，最多只有一个进行中的拼团。

* 系统中的拼团人数以当前进行中的 `group.current_count` 为准，课程本身不维护累计已拼人数。

* 用户开团时，若该课程已有进行中的拼团，则不允许创建新团，需引导用户加入。

* 拼团创建后，用户不可自行取消（需通过客服手动退款）。

* 成团自动触发：当人数达到成团人数时，立即更新状态为“已成团”，并向所有团员发送模板消息。

* 失败自动退款：倒计时结束未成团，系统自动调用微信退款接口，原路退回，并发送模板消息。

* 所有成员均可通过拼团详情页的“邀请好友”功能分享拼团。



### 5.2 退款规则

* 仅支持拼团失败自动退款，其他情况（用户主动取消、课程取消）需客服后台手动退款。

* 退款后订单状态变更为“已退款”。



### 5.3 保险说明

* 平台为每节成功开课的课程线下购买运动意外险，家长端仅作文案说明，无线上对接。

## 7. 非功能性需求



### 7.1 微信小程序适配

* 所有页面布局需遵循微信小程序设计规范，包括但不限于：

  * 页面顶部导航栏统一使用小程序默认样式或自定义导航

  * 底部tab栏使用微信原生tabBar

  * 按钮大小、点击区域符合手指触控要求（最小44pt）

  * 考虑iOS和Android设备的适配，避免使用不兼容的CSS属性

* 兼容性：支持iOS 12+ 及 Android 8.0+ 微信版本 8.0+



### 7.2 性能要求

* 页面加载时间：首屏 < 1.5秒

* 接口响应时间：< 300ms

* 支持并发用户数：初期1000人同时在线



### 7.3 安全性

* 用户敏感信息（openid、token等）需安全存储

* 支付接口需遵循微信支付安全规范

* 后台管理需登录验证，防止未授权访问

* 符合《个人信息保护法》要求，明确用户协议和隐私政策，获得用户同意后方可收集个人信息



### 7.4 可维护性

* 代码结构清晰，关键业务逻辑注释

* 后台管理界面简洁，操作流畅



### 7.5 合规性

* 协议页面应清晰可读，包含版本号和生效日期
