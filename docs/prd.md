## 1. 产品概述



### 1.1 项目背景

随着家长对儿童体能发展的重视，少儿体适能培训需求旺盛，但现有机构多需固定场地接送，耗时且成本高。深圳市场调研显示，家长对“家门口”的便利课程及拼团降价模式有强烈需求。本产品旨在连接社区家长与体适能教练，通过小区拼团、上门授课的方式，降低课程价格，解决接送难题。



### 1.2 产品定位

一款面向深圳社区家长的微信小程序，让家长在小区内发起或参与儿童体适能课程拼团，平台负责课程配置与教练协调，实现“满员即开、未成团退款”的轻量化服务。



### 1.3 核心目标

* MVP验证家长对社区拼团模式的接受度与复购率

* 跑通“运营配置课程→家长开团/参团→自动成团→线下授课”的业务闭环

* 积累种子用户，为后续迭代提供依据



### 1.4 核心规则说明

* **一个课程只能有一个进行中的拼团**：每个课程（特定时间、地点）在报名截止时间前，最多只有一个进行中的拼团。第一个开团的用户创建该拼团，后续用户只能加入此团，无法另开新团。拼团成功后或失败后，该课程不再有进行中的拼团。

* **无团长角色**：所有成员平等，均可分享拼团邀请好友。

* **无评价功能**：MVP版本不包含课后评价，相关页面已移除。

* **拼团规则透明**：在课程详情页和支付页均展示拼团规则，明确用户权益。



## 2. 目标用户与用户画像



### 2.1 目标用户群

* **职场妈妈**（30-40岁，双职工家庭，孩子3-10岁）：追求便利，节省接送时间，信任邻居推荐

* **全职妈妈**（28-38岁，注重孩子全面发展）：有社交需求，乐于组织社区活动，价格敏感但愿意为质量付费

* **带娃长辈**（60-70岁，子女双职工）：操作需简便，极度关注安全，依赖子女协助报名



### 2.2 典型用户画像



**画像1：职场妈妈——李女士（34岁，南山科技园产品经理）**

* 家庭情况：一家三口，儿子5岁，双职工，住南山区某大型小区

* 育儿痛点：工作繁忙，周末想陪孩子但又累；报过商场里的体能课，但接送来回1小时，停车难；孩子体能需加强，但兴趣班太多

* 价格敏感度：中等偏高，愿意为便利和质量付费，希望性价比高；能接受单次80-120元，包月300-500元

* 决策关键：便利性 > 价格 > 教练名气；希望就在小区内，不用开车；对教练资质有一定要求，但更看重孩子是否喜欢



**画像2：全职妈妈——王姐（38岁，福田某高端小区，二孩妈妈）**

* 家庭情况：两个女儿（8岁和4岁），丈夫经常出差，住福田中心区大型社区

* 育儿理念：注重全面发展，女儿们报了钢琴、舞蹈、英语，但户外活动少，想增加体能锻炼

* 价格敏感度：较高收入，但对价格仍理性，希望“物有所值”；愿意为优质课程支付溢价

* 决策关键：课程专业性 + 社交机会；希望课程有体系，教练懂儿童心理；同时希望孩子能跟小区玩伴一起上课，增进友谊



**画像3：年轻全职妈妈——小雅（28岁，龙岗区，宝宝2岁）**

* 家庭情况：一家三口，儿子2岁，丈夫在互联网公司工作，小雅全职带娃，住龙岗大型社区

* 育儿痛点：孩子精力旺盛，每天需要大量户外活动，但自己缺乏专业引导；想报早教班但价格高（200+元/节），且需固定时间地点；希望孩子多和同龄人互动

* 价格敏感度：对价格敏感，期望单次50-80元，包月200-300元；愿意拼团降低费用

* 决策关键：课程适龄性 + 教练亲和力；2岁宝宝需要温柔耐心的教练，课程以游戏为主；场地安全、干净



**画像4：带娃长辈——张爷爷（65岁，罗湖区，孙子6岁）**

* 家庭情况：儿子儿媳双职工，工作日主要由张爷爷接送孙子上学、带孙子户外活动，住罗湖老小区

* 育儿特点：疼爱孙子，但担心自己不懂教育，希望孙子多运动少看电视；平时带孙子在小区广场玩滑板车、追逐

* 数字能力：会用微信发消息、看视频，但不熟悉小程序操作，复杂的报名流程需要子女帮忙

* 决策关键：便利性 + 安全性：就在小区内，不用远走；有保险保障；课程时间在白天（他带孩子时段）；教练可靠

* 特殊需求：希望报名方式简单（如子女代报，或小程序有大字版/语音辅助）；课程提醒及时

## 4. 详细功能需求



### 4.1 家长端小程序



#### 4.1.1 登录/授权与用户协议同意



**触发场景**：

* 用户首次进入小程序，或未登录状态下点击需授权功能（如开团、参团、我的页面）。

* 用户主动点击“我的”页面中的登录按钮。



**交互流程**：

1. **首次进入小程序**：

   * 小程序启动后，检测用户是否已授权登录（是否有token）。

   * 若未登录，**不强制立即登录**，允许用户浏览首页和课程详情页，但点击需登录的功能时触发登录流程。

2. **点击需登录功能时**：

   * 弹出半屏或模态框，引导用户授权。

   * **步骤一：用户信息授权**：调用 `wx.getUserProfile`（根据微信最新政策）获取用户昵称和头像。同时展示《用户服务协议》和《隐私政策》链接，并默认勾选“我已阅读并同意”。若用户取消授权，则无法继续，停留在当前页。

   * **步骤二：手机号授权（可选）**：为方便联系，在用户首次登录成功后，可弹出手机号授权框（`wx.getPhoneNumber`）。用户可跳过，但建议后续在“我的”页面可补充。若用户拒绝手机号，仍可使用核心功能，但无法接收短信通知（仅能接收模板消息）。

3) **协议同意**：

   * 在用户信息授权框底部，增加复选框，文案为“我已阅读并同意《用户服务协议》和《隐私政策》”，默认选中。点击协议名称可跳转到协议详情页（H5或富文本页面）。

   * 若用户取消勾选，则“允许”或“登录”按钮置灰不可点。

4) **登录成功后**：

   * 后端返回token，前端存储，并更新用户状态。

   * 跳转回原来操作的功能（如开团页）。



**数据源**：

* 协议内容：静态富文本，可由运营后台配置（后续迭代），MVP版本直接内置在代码中或通过CMS管理。



#### 4.1.2 首页



* **顶部定位栏**：

  * 左侧显示当前定位（如“深圳南山区”），带下拉箭头图标。

  * 点击定位栏触发重新定位，调用微信的 `wx.chooseLocation` 或 `wx.getLocation` 获取新位置，更新列表距离排序。

* **顶部tab**：

  * 全部课程（默认选中）：按距离从近到远显示课程

  * 最近开课：按上课时间从近到远显示课程

* **内容区**：

  * 课程卡片列表，无限滚动加载

* **课程卡片元素及数据来源**：



| 卡片元素           | 数据来源（数据库字段/计算逻辑）                                                     |
| -------------- | -------------------------------------------------------------------- |
| 封面图            | `course.cover`                                                       |
| 课程名称           | `course.title`                                                       |
| 上课时间           | `course.start_time` \~ `course.end_time`，格式化为“MM月DD日 周X HH:MM-HH:MM” |
| 上课地点           | `course.location_community` + `course.location_detail`               |
| 拼团价            | `course.group_price`（单位：分，前端转换为元）                                    |
| 原价             | `course.original_price`（划线）                                          |
| 已拼人数           | 若该课程存在进行中的拼团（`group.status=0`），则显示该拼团的 `current_count`，否则显示0         |
| 即将成团标签         | 若该课程存在进行中的拼团且 `current_count >= target_count - 2`，则显示                |
| 距离（仅“全部课程”tab） | 根据用户经纬度与课程固定地点（需提前存储经纬度）计算，单位km                                      |
| 操作按钮“立即加入”     | 无数据源，固定文本                                                            |



* **交互**：

  * 点击卡片任意区域进入课程详情页

  * 下拉刷新列表，上拉加载更多



#### 4.1.3 课程详情页



* **页面结构**（从上到下）：

  1. **顶部轮播图**：高度200px，可左右滑动，支持点击放大。数据来源：`course.images`（JSON数组）

  2. **课程信息卡片**（白色背景，圆角，内边距16px）：

     * 课程标题（20px加粗）：`course.title`

     * 价格行：拼团价（24px橙色） 原价（16px灰色划线）：`course.group_price`、`course.original_price`

     * 成团人数：x人成团（图标+文字）：`course.target_count`

     * 上课时间、上课地点、适用年龄（每行图标+文字）：`course.start_time`、`course.end_time`、`course.location_community`、`course.location_detail`、`course.age_range`

  3) **课程介绍区**：标题“课程介绍”，下方富文本（图片+文字）：`course.description`

  4) **拼团规则说明区**（新增，位于教练信息之前）：

     * 标题：“拼团规则”（加粗，16px）

     * 内容以列表形式展示，每条前带圆点或信息图标，文案如下：

       * 每个课程仅有一个进行中的拼团，先到先得。

       * 达到成团人数自动成团，未达到则自动原路退款。

       * 所有成员均可邀请好友加入，无团长角色。

       * 拼团成功后不可退款，失败自动退款。

       * 如有疑问，请联系客服。

     * 背景色可浅灰，圆角卡片，与教练信息区分隔。

  5. **教练信息区**：

     * 标题“教练信息”

     * 教练姓名（加粗）：`course.coach_name`

     * 教练简介（灰色小字）：`course.coach_intro`

     * 证书图片缩略图（小图，可点击放大）：`course.coach_cert`（JSON数组）

  6. **保险说明区**：灰色背景区块，文案“本课程赠送运动意外险，由平台统一购买”，带保险图标。固定文案。

  7) **当前拼团状态区**：

     * **情况A（有进行中拼团）**：

       * 显示拼团摘要：当前人数/成团人数（进度条或数字），剩余时间，成员头像横排（最多6个）

       * 右侧按钮“去参团”（橙色填充）

       * 数据来源：`group`表（`course_id`=当前课程，`status=0`）

     * **情况B（无进行中拼团）**：

       * 显示灰色提示“暂无进行中的拼团，立即开团吧”

       * 右侧按钮“立即开团”（橙色填充）

  8) **底部固定栏**：

     * 左侧：客服图标（点击弹出客服二维码弹窗）

     * 右侧：按钮，与上方状态区按钮同步（“立即开团”或“去参团”）



* **特殊状态**：

  * 课程已截止报名（`deadline < now`）：底部按钮置灰，文字“已截止”

  * 课程已成团（拼团已满且`group.status=1`）：底部按钮置灰，文字“已成团”



* **交互**：

  * 点击“立即开团” → 检查是否有进行中拼团，若无则跳转开团支付页；若有则弹窗提示“该课程已有拼团，是否加入？”并提供“去参团”和“取消”选项。

  * 点击“去参团” → 跳转至该拼团详情页。

  * 点击客服图标 → 弹出客服二维码弹窗（详见4.1.7）。

  * 点击证书图片 → 放大查看。



#### 4.1.4 开团/参团支付页



* **页面结构**（从上到下）：

  1. **订单信息卡片**：

     * 课程名称：从所选课程传入

     * 上课时间、地点摘要：从所选课程传入

     * 支付金额：¥xx（拼团价，大号字体）：`course.group_price`

  2. **成团说明卡片**：

     * 文案：“满 \[成团人数] 人即开团，截止 \[报名截止时间] 未成团则自动原路退款”

     * 带信息图标

     * 数据来源：`course.target_count`、`course.deadline`

  3) **拼团规则说明区**（新增，位于协议勾选之前）：

     * 标题：“拼团规则”（加粗，14px）

     * 内容简要展示核心规则（3-4条），如：

       * 每个课程仅有一个进行中拼团。

       * 达到成团人数自动成团，未达到自动退款。

       * 所有成员均可邀请好友加入，无团长角色。

       * 拼团成功后不可退款。

  4) **协议勾选区**：

     * 复选框（默认选中）

     * 文案：“我已阅读并同意《课程服务协议》”（带下划线，点击跳转协议详情页）

  5. **底部按钮**：“确认支付”（橙色填充，受复选框控制，未勾选时置灰）



* **交互**：

  * 用户需勾选协议后方可点击确认支付。

  * 点击“确认支付”调用微信支付。

  * 支付成功 → 跳转至拼团详情页（开团则创建新团，参团则原团人数+1）。

  * 支付失败 → 提示“支付失败，请重试”。

  * 用户可返回。



* **开团特殊逻辑**：在调用支付前，后端需再次校验该课程是否仍有进行中拼团（防止并发开团），若无则创建新团并返回支付参数；若有则返回错误码，前端提示“该课程已有其他用户开团，请直接加入”。



#### 4.1.5 拼团详情页（含完整课程信息）



* **页面结构**（从上到下）：

  1. **顶部状态标签**：圆角标签，背景色对应状态（橙色进行中、绿色已成团、灰色已失败），文字如“进行中”、“已成团”、“已失败”。

  2. **课程信息卡片**（白色背景，圆角，内边距16px）：

     * 课程标题（20px加粗）

## 5. 系统规则



### 5.1 拼团规则（核心）

* **一个课程只能有一个进行中的拼团**：每个课程在报名截止时间前，最多只有一个 `status=0` 的拼团。

* 用户开团时，若该课程已有进行中的拼团，则不允许创建新团，需引导用户加入。

* 拼团创建后，用户不可自行取消（需通过客服手动退款）。

* 成团自动触发：当人数达到成团人数时，立即更新状态为“已成团”，并向所有团员发送模板消息。

* 失败自动退款：倒计时结束未成团，系统自动调用微信退款接口，原路退回，并发送模板消息。

* **无团长角色**：所有成员平等，均可通过拼团详情页的“邀请好友”功能分享拼团。



### 5.2 退款规则

* 仅支持拼团失败自动退款，其他情况（用户主动取消、课程取消）需客服后台手动退款。

* 退款后订单状态变更为“已退款”。



### 5.3 保险说明

* 平台为每节成功开课的课程线下购买运动意外险，家长端仅作文案说明，无线上对接。

## 7. 非功能性需求



### 7.1 微信小程序适配

* 所有页面布局需遵循微信小程序设计规范，包括但不限于：

  * 页面顶部导航栏统一使用小程序默认样式或自定义导航

  * 底部tab栏使用微信原生tabBar

  * 按钮大小、点击区域符合手指触控要求（最小44pt）

  * 考虑iOS和Android设备的适配，避免使用不兼容的CSS属性

* 兼容性：支持iOS 12+ 及 Android 8.0+ 微信版本 8.0+



### 7.2 性能要求

* 页面加载时间：首屏 < 1.5秒

* 接口响应时间：< 300ms

* 支持并发用户数：初期1000人同时在线



### 7.3 安全性

* 用户敏感信息（手机号、openid）加密存储

* 支付接口需遵循微信支付安全规范

* 后台管理需登录验证，防止未授权访问

* 符合《个人信息保护法》要求，明确用户协议和隐私政策，获得用户同意后方可收集个人信息



### 7.4 可维护性

* 代码结构清晰，关键业务逻辑注释

* 后台管理界面简洁，操作流畅



### 7.5 合规性

* 协议页面应清晰可读，包含版本号和生效日期
