# 邻动体适能运营后台真实库表核验报告

| 文档版本 | 修改日期       | 修改人 | 修改内容 |
| ---- | ---------- | --- | ---- |
| V1.0 | 2026-03-25 | Codex | 基于当前 Supabase 只读核验结果整理首版真实 schema 报告 |

## 1. 核验说明

本报告基于当前项目 `backend/.env` 中配置的 Supabase 连接信息，对现有数据库进行了只读核验，目的是确认运营后台设计文档与真实数据结构之间的差异，为后续后台开发提供准确的实现依据。

本次核验关注的核心对象包括：

1. `courses`
2. `groups`
3. `group_members`
4. `orders`
5. `users`
6. `admin_users`
7. `admin_log`

核验方式：

1. 逐表只读查询，确认表是否存在。
2. 读取样本记录字段，确认当前真实列集合。
3. 抽查状态字段取值，确认现有业务值域。

## 2. 总体结论

当前线上真实 schema 与 `docs/console` 设计文档存在明显差异，结论如下：

1. 现有业务核心表已经存在：`courses`、`groups`、`group_members`、`orders`、`users`。
2. 运营后台专属表当前不存在：`admin_users`、`admin_log`。
3. 真实课程表字段更偏向小程序当前实现，和后台 PRD 中的标准化字段命名并不一致。
4. 当前订单状态值为 `pending`、`success`，尚未看到后台 PRD 中的退款态。
5. 当前团状态值为 `active`、`success`，尚未看到 `failed` 样本，但代码中已支持失败态语义。

这意味着运营后台若要按现有 PRD 完整落地，需要先进行一轮“表结构补齐 + 字段映射统一 + 管理端模型建设”。

## 3. 真实表存在性核验

### 3.1 已存在表

以下表已在当前 `public` schema 中存在：

1. `courses`
2. `groups`
3. `group_members`
4. `orders`
5. `users`

### 3.2 当前不存在表

以下表在当前 `public` schema 中未找到：

1. `admin_users`
2. `admin_log`

说明：

数据库返回的是 `PGRST205`，表示在当前 schema 缓存中未发现对应表。结合本地代码情况，可以视为这两张管理端专用表尚未落地。

## 4. 真实字段核验结果

### 4.1 `courses`

当前核验到的真实字段如下：

| 字段名 | 说明 |
| ---- | ---- |
| `id` | 主键 |
| `name` | 课程名称 |
| `cover` | 封面图 |
| `address` | 上课地址，当前为单字段 |
| `start_time` | 上课开始时间 |
| `group_price` | 拼团价 |
| `original_price` | 原价 |
| `description` | 课程介绍 |
| `coach_name` | 教练姓名 |
| `coach_intro` | 教练简介 |
| `coach_certificates` | 教练证书数组 |
| `insurance_desc` | 保险说明 |
| `rules` | 规则说明 |
| `images` | 轮播图数组 |
| `age_limit` | 适龄范围 |
| `created_at` | 创建时间 |
| `service_qr_code` | 客服二维码 |
| `max_groups` | 最大成团次数 |

初步判断：

1. 当前课程表已支持部分后台所需字段，但命名与 PRD 不一致。
2. 当前缺少后台 PRD 中明确提出的多个结构化地址字段：
   `location_district`
   `location_community`
   `location_detail`
3. 当前未核验到后台 PRD 中要求的：
   `deadline`
   `longitude`
   `latitude`
   独立的课程成团人数课程字段
4. 当前存在 `max_groups`，但这表示“最大开团次数”，不是“每团目标成团人数”。

### 4.2 `groups`

当前核验到的真实字段如下：

| 字段名 | 说明 |
| ---- | ---- |
| `id` | 主键 |
| `course_id` | 课程 ID |
| `target_count` | 该团目标成团人数 |
| `current_count` | 当前成团人数 |
| `expire_time` | 拼团截止时间 |
| `status` | 团状态 |
| `creator_id` | 开团用户 ID |
| `created_at` | 创建时间 |

初步判断：

1. 目标成团人数当前落在 `groups.target_count`，不是课程表静态字段。
2. 这与后台 PRD “课程管理表单配置成团人数”的设计存在差异。
3. 如果后台后续允许配置课程成团人数，可能需要新增课程字段并在开团时写入 `groups.target_count`。

### 4.3 `group_members`

当前核验到的真实字段如下：

| 字段名 | 说明 |
| ---- | ---- |
| `id` | 主键 |
| `group_id` | 团 ID |
| `user_id` | 用户 ID |
| `joined_at` | 入团时间 |

初步判断：

1. 当前入团关系模型清晰。
2. 与交接文档一致，支付成功后才应写入该表。

### 4.4 `orders`

当前核验到的真实字段如下：

| 字段名 | 说明 |
| ---- | ---- |
| `id` | 主键 |
| `user_id` | 用户 ID |
| `course_id` | 课程 ID |
| `group_id` | 团 ID |
| `amount` | 订单金额 |
| `status` | 订单状态 |

初步判断：

1. 当前订单模型偏轻量，仅能支撑小程序现有下单与模拟支付链路。
2. 尚未看到后台 PRD 所需的以下关键字段：
   `order_no`
   `pay_time`
   `refund_time`
   `refund_reason`
3. 这会直接影响后台订单列表、订单详情、退款记录能力。

### 4.5 `users`

当前核验到的真实字段如下：

| 字段名 | 说明 |
| ---- | ---- |
| `id` | 主键 |
| `openid` | 微信 openid 或 mockOpenId |
| `nickname` | 用户昵称 |
| `avatar_url` | 用户头像 |

3. 如果后台仍要支持手机号展示，必须先确认产品是否恢复手机号采集，或调整 PRD 口径。

### 4.6 `admin_users`

1. 表不存在。

影响：

1. 管理员登录无法基于真实表建模。
2. 角色权限体系无法按文档落地。

### 4.7 `admin_log`

当前状态：

1. 表不存在。

影响：

1. 后台操作日志无法直接按文档落地。
2. 退款、删号、下架等敏感操作暂不具备审计承载表。

## 5. 状态值核验结果

### 5.1 团状态 `groups.status`

当前查询到的状态值：

1. `active`
2. `success`

说明：

1. 代码中已有 `failed` 兼容逻辑，但本次样本中未查到该状态。
2. 后台若要支持全部业务状态展示，仍应按 `active/success/failed` 三态兼容设计。

### 5.2 订单状态 `orders.status`

当前查询到的状态值：

1. `pending`
2. `success`

说明：

1. 当前订单状态尚未出现退款态。
2. 后台 PRD 中的“已支付 / 已退款”双态模型，与现有订单表并不完全对应。
3. 后续若补退款能力，需明确新增状态值，例如 `refunded`，以及对应时间字段。

## 6. 与后台 PRD 的关键差异

### 6.1 课程模型差异

后台 PRD 期待的能力：

1. 课程名称、封面图、轮播图、适龄、价格、成团人数、时间、截止时间、区域、小区、详细地址、经纬度、教练信息、富文本介绍。

当前真实课程表现状：

1. 已具备课程名称、封面图、轮播图、价格、教练信息、课程介绍、适龄、最大开团次数等基础字段。
2. 缺少截止时间、经纬度和结构化地址字段。
3. 没有明确的课程级目标成团人数字段。

结论：

课程管理后台不能直接按 PRD 全量开做，必须先决定是：

1. 补充表字段后按 PRD 实现。
2. 先按当前真实字段做一版精简后台。

### 6.2 订单模型差异

后台 PRD 期待的能力：

1. 订单号、支付时间、退款时间、退款原因、手机号、订单状态筛选。

当前真实订单表现状：

1. 只有最小下单链路字段。
2. 不足以直接支撑后台订单管理和退款审计。

结论：

订单管理模块需要先补字段和状态设计，再做后台页面。

### 6.3 管理端权限模型差异

后台 PRD 期待的能力：

1. 管理员登录。
2. 超级管理员与普通管理员角色区分。
3. 审计日志。

当前真实现状：

1. 不存在管理员表。
2. 不存在操作日志表。
3. 当前鉴权仅面向小程序用户。

结论：

管理端权限体系需要从零建设。

### 6.4 用户手机号能力差异

后台 PRD 期待的能力：

1. 订单列表展示用户手机号，便于联系。

当前真实现状：

1. 用户表没有手机号。
2. 小程序技术文档也明确当前已移除手机号授权流程。

结论：

这一点是明显的产品与技术口径冲突项，必须先统一。

## 7. 对运营后台开发的直接影响

### 7.1 可以立即推进的部分

1. 后台前端工程骨架。
2. 管理端 BFF 基础骨架。
3. 课程管理页面的现有字段版本。
4. 基于现有真实字段的课程列表和课程编辑接口设计。

### 7.2 必须先补齐再推进的部分

1. 管理员登录与角色权限。
2. 操作日志体系。
3. 订单后台列表、订单详情、退款能力。
4. 手机号展示能力。
5. 课程表单中截止时间、经纬度、结构化地址、课程级目标成团人数的正式数据方案。

## 8. 建议动作

建议按以下顺序处理：

1. 先补数据库设计：
   新建 `admin_users`、`admin_log`。
   明确是否扩展 `courses`、`orders`、`users` 表。
2. 再补接口设计：
   输出 `/api/admin/*` 接口草案。
3. 最后进入正式开发：
   先做登录和课程管理，再做订单和账号管理。

## 10. 结论

当前真实数据库结构足以支撑小程序现有课程、拼团、下单和模拟支付链路，但不足以直接支撑 `docs/console` 中定义的完整运营后台。后续开发应以本报告为基线，先完成管理端专属表与关键业务字段补齐，再进入后台功能实现阶段。否则即使页面做出来，也会因为数据承载能力不足而频繁返工。
