# 邻动体适能运营后台 - 拼团管理模块开发计划

## 1. 模块定位

拼团是课程下的子实体，不作为独立顶层业务对象存在。

运营后台新增“拼团管理”模块，目标不是替代课程管理和订单管理，而是补齐运营视角下对拼团过程的观测、排查和结果确认能力。

模块职责：

* 查看课程下所有历史拼团
* 查看单个拼团的状态、成员、订单、退款结果
* 帮助运营排查“为什么没成团 / 为什么退款 / 当前团进行到哪一步”

当前版本只做 P0、P1，不做高风险人工改数能力。


## 2. 已确认业务规则

### 2.1 实体关系

* 一个课程下可以有多个拼团
* 同一时刻仅允许一个 `active` 拼团
* 订单属于拼团，拼团属于课程

### 2.2 拼团状态

* `active`：进行中
* `success`：已成团
* `failed`：已失败

### 2.3 关键规则

* 新建拼团时，`expire_time` 统一等于课程报名截止时间
* 只有支付成功后，用户才真正写入 `group_members`
* 只有支付成功后，才更新 `groups.current_count`
* 当 `current_count >= target_count` 时，拼团变为 `success`
* 到 `expire_time` 仍未成团时，拼团变为 `failed`
* 课程成功定义为“报名截止前至少一个团成功”
* 拼团失败自动退款时：
  * 团状态改为 `failed`
  * 相关 `orders.status = refunded`
  * 保留 `group_members`，不删除
* 同一用户同一课程只能成功参团一次：
  * 仅按成功团 / 成功订单判断
  * 失败团成员记录不拦截未来再次参团


## 3. P0 功能范围

### 3.1 拼团列表页

展示字段：

* 拼团ID
* 所属课程名称
* 团状态
* 当前人数
* 成团人数
* 开团人
* 团截止时间
* 创建时间

筛选条件：

* 课程名称
* 拼团状态
* 开团时间范围
* 截止时间范围

### 3.2 课程下查看拼团

* 在课程列表或课程详情提供“查看拼团”入口
* 可查看某课程下全部历史拼团
* 页面需明确展示：
  * 当前是否存在 `active` 团
  * 历史成功团数量
  * 历史失败团数量

### 3.3 拼团详情页

基础信息：

* 拼团ID
* 所属课程
* 团状态
* 当前人数 / 成团人数
* 开团人
* 团截止时间
* 创建时间

成员信息：

* 成员昵称
* 加入时间
* 关联订单号
* 订单状态

退款信息：

* 是否已退款
* 退款时间
* 退款原因
* 退款类型（系统自动 / 手动）

### 3.4 状态只读

* 运营后台仅展示拼团状态
* 不允许人工直接修改 `active / success / failed`


## 4. P1 功能范围

### 4.1 规则透出

在拼团详情页或提示区明确展示：

* 团截止时间等于课程报名截止时间
* 课程成功定义为“报名截止前至少一个团成功”
* 失败团自动退款后保留成员历史记录

### 4.2 关联订单列表

拼团详情页增加关联订单表格，字段包括：

* 订单号
* 用户昵称
* 支付金额
* 支付状态
* 退款状态
* 退款时间
* 退款原因

### 4.3 异常排查提示

至少覆盖以下异常：

* 课程已拼团失败，但仍存在 `active` 团
* 团 `current_count` 与成员数不一致
* 失败团存在未退款订单


## 5. 当前不纳入范围

以下能力当前不开发：

* 手工修改团人数
* 手工把团改成成功
* 手工增删 `group_members`
* 脱离订单直接给用户加团
* 人工新建“空团”


## 6. 页面建议

### 6.1 拼团列表页

适合作为后台一级导航“拼团管理”的列表页，便于运营跨课程排查。

### 6.2 课程详情中的拼团记录区

适合作为课程维度查看入口，便于从课程跳转到对应拼团。

### 6.3 拼团详情页

承载成员、订单、退款结果、异常提示。


## 7. 接口建议

P0/P1 建议接口如下：

* `GET /api/admin/groups`
  用于拼团列表查询
* `GET /api/admin/groups/:id`
  用于拼团详情
* `GET /api/admin/courses/:id/groups`
  用于课程下拼团列表
* `GET /api/admin/groups/:id/orders`
  用于拼团关联订单

当前版本不建议开放：

* `POST /api/admin/groups/:id/success`
* `POST /api/admin/groups/:id/add-member`
* `POST /api/admin/groups/:id/remove-member`


## 8. 数据查询建议

核心依赖表：

* `courses`
* `groups`
* `group_members`
* `orders`
* `users`

查询原则：

* 团状态以 `groups.status` 为准
* 成员历史以 `group_members` 为准
* 支付与退款结果以 `orders.status / refund_time / refund_reason` 为准
* 同一用户是否已成功参团，以成功团 / 成功订单判断，不以失败团成员关系判断


## 9. 验收标准

* 可以按课程名称、拼团状态查询拼团列表
* 可以从课程进入查看该课程下所有历史拼团
* 拼团详情可以看到成员、订单、退款结果
* 失败团保留成员历史记录
* 失败团的退款结果能在拼团详情页正确展示
* 异常场景能被明确提示，不需要运营自己查数据库


## 10. 推荐开发顺序

1. 后端 `groups` 管理接口
2. 拼团列表页
3. 拼团详情页
4. 课程详情中的“查看拼团”入口
5. 关联订单区块
6. 异常排查提示
