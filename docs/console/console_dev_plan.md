## 1. 背景

邻动体适能运营后台是服务内部运营人员的 Web 管理系统，用于统一管理课程、订单和后台管理员账号。根据现有文档，运营后台与微信小程序共享同一个 Supabase 数据层，后台的课程配置将直接影响家长端首页、课程详情页、订单信息和报名转化链路。

当前小程序侧已经进入较明确的业务收口阶段，尤其是拼团、多团课程、支付确认和课程详情职责边界已经形成稳定口径。运营后台开发不能脱离这些既有规则单独设计，否则容易造成后台配置语义与小程序展示、订单流转、拼团统计不一致。

结合 [`console_prd.md`](/Users/yun/lindong/docs/console/console_prd.md)、[`console_api.md`](/Users/yun/lindong/docs/console/console_api.md)、[`console_tech.md`](/Users/yun/lindong/docs/console/console_tech.md) 与 [`handoff.md`](/Users/yun/lindong/docs/miniprogram/handoff.md)，本计划旨在明确运营后台的建设目标、开发范围、排期安排、关键风险与验收标准，作为后续实施和联调依据。

## 2. 目标

### 2.1 产品目标

建设一套可正式支撑日常运营的后台系统，满足以下核心诉求：

1. 支持管理员安全登录并按角色访问系统功能。
2. 支持课程新增、编辑、查询、下架，且字段与家长端展示严格一致。
3. 支持订单查询、详情查看与手动退款，保证退款操作可追踪、可审计。
4. 支持后台管理员账号管理，控制超级管理员与普通管理员权限边界。

### 2.2 技术目标

1. 基于现有 `Supabase + Express + 小程序` 数据体系扩展管理端能力，避免形成第二套数据口径。
2. 采用前后端分离方式，前端使用 React SPA，后端提供 `/api/admin/*` 管理端接口。
3. 保证课程、订单、账号等关键操作具备统一鉴权、日志记录和异常处理机制。
4. 为后续图片上传、富文本编辑、操作审计、部署上线留出清晰扩展位。

### 2.3 业务一致性目标

运营后台必须兼容当前小程序已确认口径，尤其包括：

1. 首页课程卡片展示课程历史累计成功拼团人数，而非当前团即时人数。
2. `maxGroups > 1` 时，课程详情页顶部不展示“已拼 X 人”。
3. 拼团详情页保持单团详情定位，不承接课程维度的多团聚合展示。
4. 创建订单但未支付时，不增加拼团人数，也不进入“我的拼团”。
5. 只有支付成功后，才真正入团并更新 `group_members` 与 `groups.current_count`。

## 3. 范围

### 3.1 本期开发范围

#### 3.1.1 登录认证

1. 管理员登录页。
2. Token 存储与续期控制。
3. 路由守卫与未登录跳转。
4. 基于角色的权限控制。

#### 3.1.2 课程管理

1. 课程列表展示、搜索、分页、时间筛选。
2. 新建课程。
3. 编辑课程。
4. 下架课程。
5. 课程封面、轮播图、教练证书上传。
6. 课程介绍富文本编辑。

课程字段以 PRD 和字段映射表为准，覆盖课程名称、封面图、轮播图、适用年龄、原价、拼团价、成团人数、上课时间、报名截止时间、区域、小区名称、详细地点、经纬度、教练信息、课程介绍等内容。

#### 3.1.3 订单管理

1. 订单列表展示、分页和多条件搜索。
2. 订单详情查看。
3. 已支付订单的手动退款。
4. 退款原因填写与退款结果反馈。
5. 订单查询字段与小程序订单信息保持一致。

#### 3.1.4 账号管理

1. 管理员账号列表。
2. 新增管理员账号。
3. 编辑账号角色与密码。
4. 删除普通管理员账号。
5. 超级管理员保护策略。

#### 3.1.5 基础能力

1. 统一接口封装与错误处理。
2. 操作日志记录。
3. 上传签名接口。
4. 部署环境变量与基础监控接入。

### 3.2 本期不纳入范围

以下内容不作为本期 MVP 必须项，可在后续版本扩展：

1. Banner 管理。
2. 公告管理。
3. 数据分析看板。
4. 高级权限模型，如菜单级或按钮级细粒度授权。
5. 小程序团成员明细的后台可视化运营工具。
6. 批量导入导出、批量退款等批处理能力。

### 3.3 实施边界说明

1. 后台负责课程静态信息配置和订单运营，不直接人工修改拼团人数。
2. 后台如需展示拼团相关信息，应以真实订单支付结果和团状态为准。
3. 对于小程序侧语义存在风险的接口，如 `GET /api/courses/:id/active-group`，后台不应直接按接口名称推断含义，应在管理端 BFF 中重新封装明确语义的数据查询。
4. `docs/console/邻动体适能运营后台` 目录下的示例项目可作为页面结构和交互参考，但不应直接视为正式可上线代码底座。

## 4. 排期

建议按 7 个阶段推进，先完成数据与权限骨架，再逐步闭环课程、订单、账号管理。

### 4.1 阶段一：需求与数据对齐（第 1 周）

目标：完成正式开发前的语义对齐，避免后续反复返工。

交付内容：

1. 管理端字段映射清单。
2. 管理端接口清单与命名空间约定。
3. Supabase 实际表结构核对结果。
4. 小程序现有接口与后台新接口的职责划分结果。

重点事项：

1. 核对 `course`、`groups`、`order`、`users`、`admin_users`、`admin_log` 是否与文档一致。
2. 明确退款后是否需要同步回滚拼团成员关系和团人数。
3. 明确后台课程状态字段与小程序展示状态之间的映射关系。

### 4.2 阶段二：后端管理端骨架（第 2 周）

目标：完成管理端 BFF 基础设施建设。

交付内容：

1. Express 管理端服务骨架。
2. `/api/admin/login` 登录接口。
3. 鉴权中间件、角色校验中间件。
4. 统一响应结构与错误码。
5. Supabase 管理端客户端封装。
6. 操作日志中间件初版。

### 4.3 阶段三：前端后台骨架（第 3 周）

目标：完成后台前端通用结构和基础工程能力。

交付内容：

1. React + Vite + Ant Design 项目初始化。
2. 登录页、主布局、菜单与面包屑。
3. axios 请求封装与 token 注入。
4. React Query 数据请求层。
5. 路由守卫与角色菜单控制。
6. 通用表格、筛选、分页、状态标签组件。

### 4.4 阶段四：课程管理闭环（第 4 周）

目标：完成课程配置核心链路。

交付内容：

1. 课程列表页。
2. 课程新增页。
3. 课程编辑页。
4. 课程下架操作。
5. 图片上传。
6. 富文本编辑器集成。
7. 表单校验与保存回显。

联调重点：

1. 课程配置后，小程序首页与课程详情页展示字段正确。
2. 拼团价、原价、上课时间、报名截止时间等校验逻辑正确。
3. 下架课程后，家长端不再展示。

### 4.5 阶段五：订单管理闭环（第 5 周）

目标：完成订单运营能力。

交付内容：

1. 订单列表页。
2. 订单详情弹窗或详情页。
3. 多条件搜索。
4. 手动退款弹窗。
5. 退款接口与结果回写。
6. 退款日志记录。

联调重点：

1. 后台订单字段与小程序支付结果一致。
2. 仅已支付订单允许退款。
3. 退款后订单状态、退款时间、退款原因记录完整。
4. 若退款影响拼团状态，相关回滚逻辑需被验证。

### 4.6 阶段六：账号管理与审计（第 6 周）

目标：完成账号管理和关键操作审计能力。

交付内容：

1. 账号列表页。
2. 新增账号弹窗。
3. 编辑账号弹窗。
4. 删除账号能力。
5. 超级管理员限制逻辑。
6. 课程、订单、账号关键操作日志可记录、可查询。

### 4.7 阶段七：联调、验收与上线准备（第 7 周）

目标：完成回归测试、部署和上线准备。

交付内容：

1. 前后端联调结果。
2. 核心流程测试报告。
3. 环境变量与部署文档。
4. 监控与告警配置。
5. 上线检查清单与回滚方案。

## 5. 风险

### 5.1 接口语义与现状不一致风险

当前小程序文档已明确指出 `active-group` 接口名称与实际返回语义不一致。在没有重新封装前，后台如果直接依赖该接口，容易把“最近成功团”误判为“当前进行中团”，从而导致运营判断和页面展示错误。

应对措施：

1. 管理端统一走 BFF 查询真实团状态。
2. 对“进行中团”“最近团”“成功团”使用明确字段命名。

### 5.2 文档字段与实际库表不一致风险

`console` 文档中的表名和字段名较完整，但当前小程序后端已有既存实现，可能存在命名差异、字段缺失或数据类型不一致问题。

应对措施：

1. 开发前先完成库表勘验。
2. 对差异项形成清单，不在开发中边做边猜。

### 5.3 退款与拼团状态联动风险

现有文档已明确“支付成功后才真正入团并更新人数”，但未完全展开“退款后是否回滚成员关系和团人数”的最终规则。如果退款逻辑定义不清，后台可能造成订单状态已退款但团人数仍占位的问题。

应对措施：

1. 在阶段一明确退款业务口径。
2. 在阶段五联调时覆盖支付后退款、成团后退款、重复退款等场景。

### 5.4 原型代码与正式实现偏差风险

`docs/console/邻动体适能运营后台` 示例项目字段命名与正式 PRD 存在差异，更接近演示原型而非生产实现。如果直接拿来扩写，后续重构成本会较高。

应对措施：

1. 将示例项目作为 UI 参考，不直接作为正式模型来源。
2. 正式实现以 PRD、API、实际库表为准。

### 5.5 权限与审计缺失风险

后台涉及账号删除、课程下架、订单退款等敏感操作，若没有统一鉴权和日志，会带来明显的运营风险和审计风险。

应对措施：

1. 敏感接口统一鉴权。
2. 敏感操作前后端都增加二次确认与日志记录。

## 6. 验收

### 6.1 登录与权限验收

1. 正确账号密码可以登录后台。
2. 错误账号密码无法登录，并返回明确提示。
3. 未登录状态访问任意后台页面会跳转到登录页。
4. 普通管理员不可访问账号管理页面和相关接口。
5. Token 失效后会触发重新登录。

### 6.2 课程管理验收

1. 可正常新建课程，所有必填字段校验有效。
2. 可正常编辑课程，修改内容保存后再次进入可正确回显。
3. 可正常下架课程，下架后家长端不再展示。
4. 课程封面图、轮播图、教练证书上传可用。
5. 课程介绍富文本内容可正常保存和展示。
6. 课程配置结果能正确反映到小程序首页和课程详情页。

### 6.3 订单管理验收

1. 订单列表展示正确，支持按订单号、昵称、手机号、课程名称搜索。
2. 订单详情可查看用户信息、课程信息、拼团信息、支付信息。
3. 仅“已支付”订单可发起退款。
4. 退款成功后，订单状态、退款时间、退款原因正确记录。
5. 重复退款、状态异常订单退款会被正确拦截。
6. 若定义了退款回滚拼团逻辑，则相关团状态和人数应同步正确。

### 6.4 账号管理验收

1. 超级管理员可查看、新增、编辑、删除普通管理员账号。
2. 超级管理员账号不可被误删，系统至少保留一个超级管理员。
3. 新增账号密码符合规则并安全存储。
4. 编辑账号时可选择不修改密码。

### 6.5 安全与运维验收

1. 所有后台接口均需登录后访问。
2. 敏感操作具备操作日志。
3. 前后端环境变量可独立配置开发、测试、生产环境。
4. 构建、部署、回滚流程有文档可执行。
5. 关键错误可被监控系统捕获。

### 6.6 联调验收

1. 课程配置与家长端展示字段一致。
2. 订单支付、退款与后台展示数据一致。
3. 多团课程场景下，后台不误改或误导小程序现有业务语义。
4. 后台上线后，不影响当前小程序拼团、支付、课程展示主链路。

## 7. 结论

运营后台本期建设应以“对齐现有业务口径、先打通基础闭环、逐步增强运营能力”为总原则推进。优先完成登录鉴权、课程管理、订单管理、账号管理 4 个核心模块，再补齐上传、审计、部署和监控等能力。所有实现均应以当前小程序的真实业务规则和 Supabase 数据结构为基础，避免出现后台与家长端口径分裂的问题。
