# 拼团开发经验复盘

## 1. 背景

本文档用于记录 2026-03-25 这轮拼团相关开发、联调与回归过程中暴露出的典型坑点，以及后续可复用的经验，便于后续继续迭代“多团课程”“课程详情”“拼团详情”“支付链路”时快速对照。

当前已明确的产品口径：

* 首页课程卡片展示课程历史累计成功拼团人数。
* 课程详情页若课程允许开多个团，则顶部不展示“已拼 X 人”。
* 拼团详情页保持单团详情定位，不承接课程维度的多团聚合展示。


## 2. 本轮典型坑点

### 2.1 产品口径先后不一致

本轮最容易反复修改的地方，不是代码实现本身，而是“人数口径”和“页面职责”在不同页面之间不一致：

* 首页一度展示当前团人数，后来切成历史累计成功人数。
* 课程详情页顶部人数一度与首页口径一致，后来又因为多团课程改成隐藏。
* 拼团详情页本质上应该是单团详情，但课程详情页又开始承接“所有团列表”。

**问题本质**：
在“课程维度”和“单团维度”没有彻底拆开的情况下，任何一个字段都可能被多个页面以不同方式解释，最终导致看起来像 bug，实际是定义不一致。


### 2.2 接口名称与接口语义不一致

`GET /api/courses/:id/active-group` 这个接口名看起来像“只返回进行中的团”，但当前实现中在没有进行中团时，会回退返回最近一个成功/失败团。

这会带来两个问题：

* 前端看到 `active-group` 名称时，容易默认把返回值当成“当前活跃团”。
* 按钮逻辑、倒计时逻辑、页面状态逻辑会因此误判，比如误把最近成功团当成“当前状态”。

**经验**：
如果接口语义已经变化，要么改接口名，要么明确增加字段标识，不要继续沿用旧名字让前端猜。


### 2.3 mock fallback 掩盖真实问题

课程列表接口最开始允许在真实请求失败时自动 fallback 到 mock 数据。结果是：

* 页面仍然正常显示
* 数据却不一定来自真实后端
* 排查“为什么首页人数不对”时，会被假数据误导

典型表现：

* 数据库里没有 `current_count = 18` 的团
* 首页却显示了 `18`
* 后来排查确认，优先怀疑的就不是数据库，而是 mock fallback

**经验**：
对首页、详情页、订单、支付这类核心链路，默认不应悄悄 fallback 到 mock。联调阶段“明确报错”比“给假数据但看起来正常”更有价值。


### 2.4 多团能力已支持，但业务阶段未完全启用

这次代码已经开始支持：

* `maxGroups`
* `completedGroupsCount`
* `successJoinedCount`
* `groupList`

但业务上前期其实还是围绕“单课程单进行中团”在跑。这样就容易出现：

* 技术能力已经多团化
* 页面逻辑开始多团化
* 实际业务运营却还没完全按多团使用

最终会导致页面规则变复杂，甚至比当前业务需要的还复杂。

**经验**：
当“未来要支持”与“当前真正启用”不一致时，页面逻辑先按当前业务阶段收口，数据结构再为未来留扩展点。不要一上来把所有页面都按未来形态改满。


### 2.5 支付成功才算入团，必须全链路统一

这次一个关键修复是：

* 创建订单时不提前增加人数
* 支付成功后才真正入团并更新人数

但这类修复不能只改一处，否则会出现新的错位：

* 首页人数先变了
* 课程详情人数没变
* “我的拼团”提前出现记录
* 支付页/结果页链路出现假成功

**经验**：
“支付成功才算成功参团”是一个跨接口、跨页面的系统规则，必须同步检查：

* 创建订单接口
* 支付成功接口
* 首页列表
* 课程详情
* 拼团详情
* 我的拼团


### 2.6 文档和代码容易再次分叉

今天最容易出现的另一类问题是：

* 代码已经按新口径改了
* PRD 里还保留旧描述
* 回头看文档时，又会误以为当前代码错了

**经验**：
只要涉及以下任意变化，就应该当天回填 PRD 或专项说明：

* 首页人数统计口径变化
* 课程详情主按钮判断变化
* 拼团详情页职责变化
* 支付链路跳转变化
* 协议页面拆分变化


## 3. 本轮沉淀出的经验

### 3.1 先统一业务口径，再写接口和页面

建议每次遇到这类功能，先用一句话定清楚每个页面的职责：

* 首页看什么维度
* 课程详情看什么维度
* 拼团详情看什么维度

如果这一步不先做，后面代码再怎么改，都容易陷入“这个页面到底该显示哪个人数”的循环。


### 3.2 课程维度与单团维度必须明确拆开

后续凡是涉及拼团展示，都建议先判断是哪个维度：

* **课程维度**：
  首页、课程详情、课程运营配置

* **单团维度**：
  拼团详情、拼团分享、拼团成员、单团进度

如果课程维度页面开始展示单团细节，一定要明确是“当前团”“最近团”还是“所有团聚合”，不能含糊。


### 3.3 宁可多返回字段，也不要让前端猜

这次比较稳定的部分，都是后端明确补了字段后才收住的，例如：

* `maxGroups`
* `completedGroupsCount`
* `successJoinedCount`
* `groupList`
* `userJoined`

**经验**：
当页面开始出现复杂分支时，优先加清晰字段，而不是让前端通过已有字段组合推断业务状态。


### 3.4 核心入口不建议默认 mock 降级

建议保留 mock 的场景：

* 独立演示
* 离线开发
* 非核心边缘页

建议禁用默认 mock 降级的场景：

* 首页课程列表
* 课程详情
* 下单接口
* 支付结果链路


### 3.5 接口级回归很有价值，但不能替代页面回归

本轮接口级回归帮助我们确认了：

* 首页列表返回字段是否正确
* 课程详情是否返回多团数据
* 非成员访问单团详情是否仍被权限拦截

但它不能替代页面层面的真实回归，例如：

* 按钮是否真的隐藏
* 多团时顶部人数是否不显示
* 分享落地页是否还是单团详情
* 倒计时文案是否正常

**经验**：
接口级回归负责验证“数据与规则”，页面回归负责验证“展示与交互”，两者要拆开看。


## 4. 后续开发建议

### 4.1 优先保持这三条口径稳定

当前建议不要再频繁改动以下三条：

* 首页课程卡片展示历史累计成功拼团人数
* 课程详情页多团时不展示顶部“已拼 X 人”
* 拼团详情页保持单团详情定位

这三条一旦稳定，后面多团能力扩展会清晰很多。


### 4.2 建议后续补一个“课程维度状态摘要”接口

如果后面课程维度规则继续复杂，建议单独抽一个课程摘要模型，例如：

* 当前是否存在进行中团
* 当前团摘要
* 历史累计成功人数
* 已成功团数
* 是否还能开新团
* 推荐按钮文案

这样首页和课程详情就能共享一套后端判断结果。


### 4.3 建议后续清理历史文档残留口径

当前 PRD 主体模块已更新，但其他文档里仍可能残留旧口径，例如：

* `docs/development-plan.md`
* `docs/regression-checklist.md`
* `docs/tech.md`

建议后续做一轮“拼团口径全量一致性清扫”。


## 5. 一句话总结

本轮最核心的经验是：

**先统一业务口径，再统一接口语义，最后才是页面实现；如果顺序反过来，就会不断修复那些看起来像 bug、实际是定义不一致的问题。**
