# 邻动体适能小程序 开发计划（最终执行版）

## 1. 总体说明

本文档为当前项目**最终采用的开发计划**，已结合最新 [PRD](/Users/yun/lindong/docs/prd.md) 与 [技术设计文档](/Users/yun/lindong/docs/tech.md) 同步更新。

后续所有代码开发、联调、文档更新，均以本计划中的阶段顺序与模块定义为准。

当前项目采用以下原则推进：

* 优先完成小程序主链路闭环：协议同意 -> 登录 -> 浏览课程 -> 查看详情 -> 开团/参团 -> 支付结果 -> 拼团详情 -> 我的拼团
* 前端开发坚持“真实接口优先，网络不可用时降级 mock”
* 页面优先保证可运行、可联调、可演示，再逐步替换为真实后端能力
* 在每个阶段结束时同步检查页面结构、交互链路与文档一致性

## 3. 分阶段开发计划

### 阶段一：基础框架与权限

#### 1.1 项目初始化与公共组件

**目标**：
完成微信小程序基础工程搭建，建立可持续扩展的页面、样式、工具和组件基础。

**工作项**：

* 初始化小程序项目结构
* 配置 `app.json`，保留底部 TabBar：首页、我的
* 封装网络请求 `utils/request.js`，统一 `baseURL`、token 注入、错误处理
* 建立通用工具函数 `utils/util.js`
* 建立全局样式 `app.wxss`
* 创建首页和我的页面基础文件
* 创建基础公共组件：加载态、空状态、通用弹窗

**当前状态**：已完成

#### 1.2 用户协议与隐私政策展示

**目标**：
在用户进入小程序前完成协议同意闭环，确保协议先于业务页面展示。

**工作项**：

* 创建独立协议页面
* 展示《用户协议》《隐私政策》富文本内容
* 首次打开小程序时强制展示协议页
* 用户必须勾选同意后才能进入首页
* 用户拒绝时提示“您需要同意协议才能使用本小程序”
* 环境支持时尝试退出小程序，不支持时停留在协议页
* 在 `app.js` 中统一管理协议同意状态

**当前状态**：已完成

#### 1.3 微信授权登录

**目标**：
完成小程序基础登录能力，仅保留头像、昵称与 token 的最小可用方案。

**工作项**：

* 实现微信登录：`wx.getUserProfile + wx.login`
* 调用后端登录接口换取 token
* 将头像、昵称、token 存储到本地缓存与 `app.globalData`
* 处理用户拒绝授权的提示逻辑
* 保留接口失败时的 mock 登录降级

**调整说明**：

* 原计划中的“手机号授权”已移除，不再申请 `wx.getPhoneNumber`
* 不再在全局状态中维护手机号字段

**当前状态**：已完成

### 阶段二：课程展示与详情

**工作项**：

* 顶部定位栏显示当前城市/区域
* 支持点击重新定位

* 展示课程卡片：封面、名称、时间、地点、拼团价、原价、已拼人数、距离、即将成团标签
* 支持下拉刷新和上拉加载更多
* 定位失败时降级使用默认坐标

**当前状态**：已完成

**当前口径补充**：

#### 2.2 课程详情页

### 阶段三：拼团与支付

## 4. 相比原计划的主要调整

### 4.1 权限与协议

* 协议页从“登录流程附属能力”提升为“小程序启动前置拦截”
* 用户必须先同意协议，才能进入首页

### 4.2 登录能力

* 删除手机号授权
* 登录流程只保留头像、昵称、token
* 简化用户体系，降低联调复杂度

### 4.3 课程与拼团展示

* 首页和详情页的“已拼人数”统一以当前进行中的拼团人数为准
* 课程不再维护历史累计拼团人数，人数统一来自 `groups.current_count / target_count`
* 课程详情页中部状态模块去掉操作按钮，避免和底部操作重复

### 4.4 支付链路

* 原计划中直接调微信支付，当前改为模拟支付
* 新增独立支付结果页，统一管理支付后的成功/失败跳转
* 支付页文案与结构统一收敛为单一“拼团说明”模块
* 模拟支付成功后需回调后端接口更新订单状态与拼团人数

### 4.5 个人中心

* 删除子女管理占位
* 删除个人中心和更多服务两个非核心模块
* “我的”页只保留核心入口：登录、退出登录、我的拼团、联系客服

### 4.6 联调策略

* 将“真实接口优先，网络不可用时才降级 mock”作为固定策略
* 对鉴权错误和服务端业务错误不再静默切 mock，优先暴露真实问题便于排查

## 5. 后续执行要求

后续所有代码生成与模块继续开发时，必须遵循以下约束：

1. 严格按照本计划阶段顺序推进，除非文档明确更新。
2. 新增页面、接口、交互时，需同步检查 [PRD](/Users/yun/lindong/docs/prd.md) 和 [技术设计文档](/Users/yun/lindong/docs/tech.md) 是否需要更新。
3. 若真实后端暂未完成，前端必须提供 mock 降级。
4. 若产品逻辑发生变化，应先同步文档，再继续实现。
